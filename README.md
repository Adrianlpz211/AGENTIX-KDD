## The problem

Every AI coding tool today has the same fundamental flaw: **the agent forgets everything when you close the chat.**

Session 1: the agent explores your codebase, learns that touching `auth.ts` breaks the session manager, figures out your conventions, understands your architecture.  
Session 2: it knows nothing. It explores again. It makes the same mistake.

This isn't a model problem. It's an architecture problem. No amount of context window size fixes a system that starts from zero every time.

**Agentic KDD solves this at the infrastructure level.** Not with a bigger context window ‚ÄĒ with persistent memory that lives inside your project, learns from every cycle, and gets smarter over time.

---

## What is KDD?

**Knowledge-Driven Development** is the methodology at the core of Agentic KDD.

Traditional AI development treats each session as independent. KDD treats knowledge as a first-class artifact: every pattern discovered, every error resolved, every architectural decision made is stored in a structured knowledge graph and fed back into every future cycle.

The agent doesn't re-learn your project. It already knows it.

```
Without KDD:
  Session 1 ‚Üí agent learns ‚Üí session ends ‚Üí forgotten
  Session 2 ‚Üí start over ‚Üí same mistakes ‚Üí same time lost

With KDD:
  Session 1 ‚Üí agent learns ‚Üí stored in .agentic/memoria.db
  Session 2 ‚Üí agent already knows ‚Üí builds on top of everything
  Session N ‚Üí the system is smarter than it was on day one
```

This is why KDD isn't a wrapper around Claude. It's a methodology for making AI-assisted development actually compound over time.

---

## How it works

Agentic KDD lives inside your project as a `.agentic/` directory ‚ÄĒ a local SQLite database, a set of specialized agents, and a pipeline that runs automatically when you type `aa:`.

```bash
aa: implement JWT authentication with refresh token rotation
```

That single command triggers a pipeline that runs without interrupting you:

```
‚Ď† Analyst      ‚ÄĒ reads memory, AST graph, ADRs, specs, plans all phases
‚Ď° Build        ‚ÄĒ frontend / backend agents implement within the plan
‚ĎĘ TDD Gate     ‚ÄĒ runs tests, parses output, self-heals up to 3√ó automatically
‚Ď£ QA           ‚ÄĒ verifies acceptance criteria against the full test suite
‚Ď§ ag:review    ‚ÄĒ automatic code review against accumulated KDD memory
‚Ď• Memory       ‚ÄĒ syncs graph, causal edges, specs, observability
```

**The developer never types `ag: test` or `ag: review`. They run automatically.**

The critical difference from every other agent framework is in step ‚ĎĘ. The TDD gate is a deterministic check in Node.js code ‚ÄĒ not a prompt suggestion. The agent cannot declare tests passing without actually running them. The gate rejects progress without proof.

```
Cursor rules:   "prefer running tests before delivering"
                ‚Üí the agent follows this when it feels like it

Agentic KDD:    if (tests_passing === false) { STOP("gate rejected") }
                ‚Üí the code rejects progress without proof of compliance
```

## Memory architecture

Four persistent memory layers, fully offline, in SQLite, inside your project.

```
.agentic/memoria.db
‚Ēā
‚Ēú‚ĒÄ‚ĒÄ WORKING MEMORY      Active session buffer
‚Ēú‚ĒÄ‚ĒÄ EPISODIC MEMORY     Raw history of what happened in every cycle
‚Ēú‚ĒÄ‚ĒÄ SEMANTIC MEMORY     Project entity graph, AST, module relationships
‚ĒĒ‚ĒÄ‚ĒÄ PROCEDURAL MEMORY   Patterns, errors, decisions ‚ÄĒ rules that apply every cycle
```

Every memory node carries a confidence signal updated automatically:

```
LOW    ‚Üí suggestion, not enforced
MEDIUM ‚Üí applied and mentioned in every plan
HIGH   ‚Üí fixed rule, applied on every cycle without exception

Applied ‚Č• 3√ó  +  utility ‚Č• 70%  ‚Üí  auto-promoted to MEDIUM
Applied ‚Č• 7√ó  +  utility ‚Č• 80%  ‚Üí  auto-promoted to HIGH
Unused for 60 cycles             ‚Üí  auto-degraded (temporal decay)
```

Beyond the four CoALA layers, Agentic KDD adds three knowledge structures specific to software development:

- **AST Graph** ‚ÄĒ symbols, imports, call graph, PageRank across 12 languages
- **Causal Edges** ‚ÄĒ `caused_failure`, `was_fixed_by`, `tested_by`, `regressed_by` ‚ÄĒ bi-temporal, never deleted, only invalidated
- **Knowledge Docs** ‚ÄĒ ADRs, gotchas, conventions ingested from `docs/adr/`

## The five phases

### Phase 0 ‚ÄĒ Harness

Deterministic enforcement gates. The piece missing from every other agent framework.

`harness.cjs` runs PRE/EXEC/POST checks on every pipeline step. `tdd-gate.cjs` is a Node.js loop that detects your test command, runs it, parses the output (Jest, Vitest, Mocha, pytest), and retries up to three times automatically. The agent cannot lie about test results.

### Phase 1 ‚ÄĒ Discernment

Before planning any change, the agent has a complete structural map of the project. `ast-indexer.cjs` builds an AST graph across 12 languages with PageRank scoring. `causal-edges.cjs` tracks what broke what, what fixed what, what tests cover what ‚ÄĒ bi-temporally, with full history preserved.

### Phase 2 ‚ÄĒ Knowledge Base

`adr-ingestor.cjs` parses Architecture Decision Records in MADR format and converts them to typed graph edges ‚ÄĒ no LLM required. `knowledge-ingestor.cjs` ingests gotchas, conventions, and runbooks. The agent understands not just what the code does, but why it was built that way.

### Phase 3 ‚ÄĒ Autonomy

`spec-manager.cjs` implements Kiro-style specs with wave execution. Wave 1 contains tasks with no dependencies. Wave N depends on Wave N-1. The agent executes the right wave every time. `impact-analyzer.cjs` runs a pre-change impact analysis combining AST structure, causal memory, and the knowledge base before touching anything ‚ÄĒ returning CRITICAL, MEDIUM, or LOW severity.

### Phase 4 ‚ÄĒ Collaborative Mode

`collab-manager.cjs` syncs the local SQLite memory to a shared Turso database. Multiple developers share one knowledge base. What one developer learns, the agent knows for everyone.

```
Dev A discovers that touching auth.ts breaks session.ts
                        ‚Üď auto-sync to Turso
Dev B already knows it before touching auth.ts
Dev C joins the team ‚Üí gets 6 months of project knowledge from day one
```

Team members join with a single invite code generated by `akdd collab invite` ‚ÄĒ no credentials to share manually, no Turso account required for team members.

## Memory governance ‚ÄĒ MemCurator

`mem-curator.cjs` is an autonomous governance agent that runs automatically every 10 cycles. It handles what no other agent framework addresses explicitly:

- **TTL enforcement**: episodic memory older than 30 days is compressed into summaries and archived after 90 days
- **Semantic deduplication**: nodes with Jaccard similarity above 0.92 are merged, preserving the higher-confidence version
- **Conflict resolution**: when new code supersedes an old rule, the old rule is marked `HISTORICO` with an explicit supersession link ‚ÄĒ not deleted
- **Relevance scoring**: every node is scored continuously using `S(k) = similarity √ó exp(-őĽ √ó őĒt) √ó log(1 + usage_count)`
- **Node limit**: active procedural memory is capped at 1000 nodes; low-utility candidates are deprecated automatically

The MemCurator is intentionally decoupled from the development agents. It has no interest in preserving its own past decisions ‚ÄĒ it only governs quality.

## Embeddings

The default embedding model is `jina-embeddings-v2-base-code` ‚ÄĒ a bimodal NL-PL model trained specifically on code. It understands type relationships, AST structures, and control flow semantics. `all-MiniLM-L6-v2` is available as a lightweight fallback (~23MB vs ~500MB) for environments where storage is constrained.

```bash
akdd jina-install    # bimodal NL-PL model ‚ÄĒ recommended
akdd embed-install   # all-MiniLM-L6-v2 ‚ÄĒ lightweight fallback
```

## Discoverability

Every `akdd sync` generates three files automatically:

- `.agentic/llms.txt` ‚ÄĒ structured project map for external agents. Architecture, active rules, known pitfalls, available MCP tools. The equivalent of `robots.txt` but for LLMs.
- `.agentic/llms-full.txt` ‚ÄĒ expanded version with all active patterns, causal memory, and ADRs.
- `.agentic/knowledge-graph.json` ‚ÄĒ the full causal graph serialized for Git versioning. The graph travels with the repository.

## Installation

```bash
npm install -g agentic-kdd
cd my-project
akdd init
```

`akdd init` detects your stack, downloads the latest files from GitHub, configures the MCP server in Cursor automatically, and optionally registers with Claude Code. Open the project in your IDE and run:

```
aa: configure
```

The system maps your codebase once. After that, every `aa:` cycle builds on accumulated knowledge.

**Requirements:** Node.js 18+, Cursor or Claude Code (or any MCP-compatible client).

## Audit Department

A separate QA system that runs deep project-wide audits, independent from the `aa:` development pipeline. Where `aa:` validates individual tasks, the Audit Department reviews the project as a whole.

```
audit: auditar     # full audit ‚ÄĒ all 7 agents in parallel
audit: seguridad   # security only ‚ÄĒ secrets, auth, vulnerabilities
audit: frontend    # frontend only ‚ÄĒ source maps, keys, build artifacts
audit: backend     # backend only ‚ÄĒ endpoints, validation, exposed APIs
audit: datos       # database only ‚ÄĒ schema, RLS, query exposure
audit: performance # performance only ‚ÄĒ rate limiting, cache, scalability
audit: browser     # real browser QA
audit: codigo      # code quality and Git hygiene
audit: help        # show all available commands
```

Audits run without touching code. Output goes to `_output/audit-[date].md`.

The 7 specialized agents:

| Agent | Focus |
|-------|-------|
| `00-director` | Orchestrates the full audit, consolidates reports |
| `01-seguridad` | Exposed secrets, auth weaknesses, injection risks |
| `02-frontend` | Build artifacts, source maps, client-side key exposure |
| `03-backend` | Endpoint security, input validation, API surface |
| `04-datos` | Database schema, RLS policies, query exposure |
| `05-performance` | Rate limiting, caching strategy, scalability bottlenecks |
| `06-browser` | Real browser QA against the running application |
| `07-codigo` | Code quality, dead code, Git history hygiene |

## Pro agents

Beyond the core pipeline, four specialized agents handle operations that run independently:

| Agent | Invocation | What it does |
|-------|------------|--------------|
| `ag:review` | `ag: review <file>` | Code review against KDD memory. Also runs automatically in every `aa:` cycle. |
| `ag:test` | `ag: test <module>` | Generates test suites based on error history and causal edges for the module. |
| `ag:refactor` | `ag: refactor <file>` | Refactors with pre-change impact analysis. Won't proceed if impact is CRITICAL. |
| `ag:doc` | `ag: doc <module>` | Technical documentation generated from code + project memory. |

## What Agentic KDD is not

It is not a chat interface, an IDE, or a replacement for Git. It does not require an internet connection after setup. It does not send your code anywhere. It does not depend on a SaaS backend.

It is infrastructure. It runs inside your project, it travels with your repository, and it compounds over time.
